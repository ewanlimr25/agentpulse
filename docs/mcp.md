# MCP-native Observability

AgentPulse ships first-class observability for Model Context Protocol (MCP)
workflows, plus an **AgentPulse MCP server** that lets IDE-side agents
(Claude Code, Cursor, Windsurf) query traces, runs, and budgets directly.

This doc covers:

1. [Span semantic conventions](#span-semantic-conventions) — the attributes a
   client SDK or MCP server sets to make a stitched cross-process trace.
2. [The AgentPulse MCP server](#the-agentpulse-mcp-server) — the binary you
   run alongside your IDE.
3. [OTLP collector Bearer auth](#otlp-collector-bearer-auth) — tightening
   the receiver-level auth without breaking existing deployments.

---

## Span semantic conventions

AgentPulse uses three MCP span kinds:

| `agentpulse.span_kind` | Side | Meaning |
|---|---|---|
| `mcp.tool_call` | client | Agent (or wrapping framework) invoked a tool *via* MCP. |
| `mcp.list_tools` | client | Agent asked the MCP server which tools are available. |
| `mcp.server` | server | The MCP server-side execution of a tool (paired with the client's `mcp.tool_call`). |

### Correlation attributes

Set the same value on both sides to stitch the cross-process trace.

| Attribute | Description |
|---|---|
| `agentpulse.mcp.session_id` | MCP session identifier; generated by the client when the session opens. |
| `agentpulse.mcp.request_id` | Per-request JSON-RPC `id` (one-shot correlation). |
| `agentpulse.mcp.client_name` | Identifies the client, e.g. `claude-code`, `cursor`. |
| `agentpulse.mcp.transport` | `stdio` \| `sse` \| `http`. |
| `agentpulse.mcp.server_name` | Logical name of the MCP server. |
| `agentpulse.mcp.tool_name` | Tool the client invoked (also mirrored to `tool.name`). |
| `agentpulse.mcp.input_schema` | JSON schema of the tool input (optional). |
| `agentpulse.mcp.output_schema` | JSON schema of the tool output (optional). |
| `agentpulse.mcp.tool_count` | Number of tools returned by `mcp.list_tools`. |
| `agentpulse.mcp.discovered_tools` | Comma-separated list of tool names returned. |

The collector also accepts the upstream `mcp.session.id`, `mcp.request.id`,
`mcp.client.name`, `mcp.transport`, `mcp.server.name`, `mcp.tool.name` keys
and normalizes them to the `agentpulse.*` form. See
[`config/agent_attributes.yaml`](../config/agent_attributes.yaml) for the
complete extraction rules.

### Python SDK

```python
from agentpulse import (
    mcp_tool_call_ctx,   # client side
    mcp_server_ctx,      # server side
)

# IDE / agent side
with mcp_tool_call_ctx(
    server_name="filesystem-server",
    tool_name="read_file",
    session_id="sess-42",
    request_id="req-7",
    client_name="claude-code",
    transport="stdio",
):
    result = mcp_client.call_tool("read_file", {"path": "/etc/hosts"})

# MCP server side — uses the SAME session_id/request_id
with mcp_server_ctx(
    server_name="filesystem-server",
    tool_name="read_file",
    session_id="sess-42",
    request_id="req-7",
    transport="stdio",
):
    contents = open("/etc/hosts").read()
```

### TypeScript SDK

```typescript
import { mcpToolCall, mcpServer } from '@agentpulse/sdk'

await mcpToolCall(
  {
    serverName: 'filesystem-server',
    toolName: 'read_file',
    sessionId: 'sess-42',
    requestId: 'req-7',
    clientName: 'claude-code',
    transport: 'stdio',
  },
  () => mcpClient.callTool('read_file', { path: '/etc/hosts' }),
)

// In the MCP server process
await mcpServer(
  {
    serverName: 'filesystem-server',
    toolName: 'read_file',
    sessionId: 'sess-42',
    requestId: 'req-7',
  },
  () => fs.readFileSync('/etc/hosts', 'utf8'),
)
```

## The AgentPulse MCP server

The `agentpulse-mcp` binary is a Model Context Protocol server (JSON-RPC 2.0
over stdio) that lets a coding agent self-diagnose runs without leaving the
IDE.

### Installation

```bash
go install github.com/agentpulse/agentpulse/backend/cmd/agentpulse-mcp@latest
```

(Or build from source: `cd backend && go build -o agentpulse-mcp ./cmd/agentpulse-mcp`.)

### Claude Code configuration

Add to `~/.claude.json`:

```json
{
  "mcpServers": {
    "agentpulse": {
      "command": "agentpulse-mcp",
      "env": {
        "AGENTPULSE_BASE_URL": "http://localhost:8080",
        "AGENTPULSE_INGEST_TOKEN": "ap_live_xxxxxxxxxxxxxxxx",
        "AGENTPULSE_PROJECT_ID": "00000000-0000-0000-0000-000000000000"
      }
    }
  }
}
```

Restart Claude Code; the new tools appear under `agentpulse.*`.

### Tools exposed

| Tool | Use it for |
|---|---|
| `search_traces` | Full-text search across spans for a project. |
| `get_run_details` | Fetch a run's metadata + spans. |
| `compare_prompts` | Diff prompts/completions between two runs of the same flow. |
| `replay_run` | Get the replay-bundle URL + a ready-to-paste `apctl replay` command. |
| `current_budget_status` | Active budget rules + recent budget alerts. |

Every tool defaults to `AGENTPULSE_PROJECT_ID`; pass `project_id` explicitly
to override.

### Example prompt

> *"Why did my last run cost \$2? Use agentpulse to dig into the highest-cost
> spans and show me what model was called."*

The agent will call `current_budget_status`, then `search_traces` with the
recent run's `run_id`, and surface the answer inline.

## OTLP collector Bearer auth

The team-mode collector now ships with the
[`agentpulsebearerauth`](../collector/extension/agentpulsebearerauth)
extension. It validates `Authorization: Bearer <token>` against the
`project_ingest_tokens` Postgres table — the same table the REST API uses.

### Default behaviour: warn-only

Out of the box, `required: false`:

- Missing or invalid Bearer headers are **logged** but the request is still
  forwarded to the pipeline. Operators can audit `agentpulsebearerauth: warn-mode reject`
  log lines to see who would be rejected once enforcement is on.

This means upgrading the collector causes **no behaviour change** for
deployments that don't set Bearer headers yet.

### Enforcing Bearer auth

Set `AGENTPULSE_OTLP_REQUIRE_BEARER=true` (or edit `collector/config.yaml`
directly):

```yaml
extensions:
  agentpulsebearerauth:
    dsn: "postgres://…"
    required: true
    cache_ttl: 60s

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: "0.0.0.0:4317"
        auth:
          authenticator: agentpulsebearerauth
      http:
        endpoint: "0.0.0.0:4318"
        auth:
          authenticator: agentpulsebearerauth

service:
  extensions: [health_check, agentpulsebearerauth]
```

Once `required: true`, the receiver returns `401 Unauthorized` on bad
tokens before any pipeline cycles are spent.

### How the SDK sends Bearer tokens

Set the OTLP exporter `Authorization` header to your project's ingest token.
Example (Python):

```python
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter

OTLPSpanExporter(
    endpoint="https://otel.your-host/v1/traces",
    headers={"Authorization": "Bearer ap_live_xxxxxxxxxxxxxxxx"},
)
```

Indie-mode users get the same enforcement automatically — the embedded OTLP
receiver in `backend/internal/otlp/receiver.go` validates Bearer tokens
unconditionally.

### Migration path

1. Deploy the new collector with `required: false` (default).
2. Update SDKs to send the Bearer header.
3. Tail the collector logs for `agentpulsebearerauth: warn-mode reject`. When
   the rate is zero, flip `AGENTPULSE_OTLP_REQUIRE_BEARER=true`.
4. The legacy `agentpulse.ingest_token` span attribute path
   (`authenforceproc`) remains in place as a transitional fallback. Remove
   it from your pipeline once `agentpulsebearerauth` is enforced.
